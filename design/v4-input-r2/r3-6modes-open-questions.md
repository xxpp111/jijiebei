## P0 · 阻塞与核心归类

### O1. 「极难=12 因子」内部判定走 std12(mfc==4) 还是 modeIsVeryHard 守卫？
- **涉及**：极难=12因子（std12）
- **背景**：R3 已拍板「12因子=极难、删极难②」。但代码现存 3 种"极难"标志（modeIsVeryHard / modeIsVeryHard2 / modeFeiqiu）都走 hard 抽取；JijieContro 混用 modeIsVeryHard 与 mfc==4。R3 合并口径下，内部应以哪个为单一判定？
- **候选**：
  - A. 纯 `mfc==4` 判定（当前末场特判即此），modeIsVeryHard 系列废弃/不上 home。
  - B. 加 `modeIsVeryHard==false` 守卫区分，保留 modeIsVeryHard 旧路径。
- **推荐**：**A**——与 R3「合并、删极难②」一致，末场强制大因子已由 `i==count-1 && mfc==4` 驱动，modeIsVeryHard①（双打挑战旧路径，含 popMap 营救矿工）若不上 home 则废弃。需确认 modeIsVeryHard①/② 是否最终下线。

### O2. 非酋「单刷实现 vs 双打归类」如何收敛？
- **涉及**：非酋（feiqiu）、官突plus
- **背景**：R3 把非酋归「双打挑战」类，但代码 mfc=1 完全是单刷链路（A/B 分组、popFactor、manualSlots），与双打独立骨架 JJBDoubles 无关。非酋在比赛中是单人应战还是 2v2？MODES 数组缺席、靠 InitPanel hardcode 入口。
- **候选**：
  - A. 非酋是「单刷的特殊难度变体」，分类挪出双打、归单刷·特殊（或独立隐藏模式）。
  - B. 非酋确为双打玩法，需重写为 JJBDoubles 类骨架（重构成本高）。
  - C. 维持归类不变，仅作 UI 分组标签，实现仍单刷，BP/难度按单刷处理。
- **推荐**：**A 或 C**——实现已是单刷，重写代价大。请确认非酋实际比赛结构（单人/双打），据此定 BP 与额外难度适用口径。归类一旦定，更新 mode-rules-truth-table 主表（补非酋行）+ MODES 数组（补入或标隐藏）。

### O3. 官突 plus 段 3④（Home→Select 入口）何时实现、范围多大？
- **涉及**：官突plus
- **背景**：骨架已完成（start/reset/matches/判定），但 Home 入口被 `soon=true` 拦截，jjbSession 无 startDoublesSession，Select/Battle 屏未适配 doubles 数据模型。
- **候选**：
  - A. 本轮实现段 3④：删 soon 拦截 + 新增 startDoublesSession + Select/Battle 适配 matchVM(doubles=true)。
  - B. 仅出 UI 设计，工程实现下一轮排期。
- **推荐**：**B（设计先行）**——本稿是设计交付轮，工程入口链路由派发决策单独排优先级。请确认是否本轮就要工程入口，还是只要设计稿。

## P1 · 落地前需定

### O4. 拯救模式 BP 启用时「自选 1 指挥官」分支是否适用？
- **涉及**：拯救（rescue）
- **背景**：拯救 7 人固定池、selfShow=false，无「自选指挥官」语义。若 BP 开启二选一，自选分支落地不了。
- **候选**：
  - A. 物理禁用：拯救 BP 仅开放「ban 因子」分支，自选分支灰掉/不显示。
  - B. 留作静默无效选项（点了无效果）。
- **推荐**：**A**——明确禁用更清晰，避免选手困惑。R3 中拯救 BP 默认关，影响面小，但开启时需此规则。同时定 ban 因子数上限是否限于池大小(=7)。

### O5. 各模式 selfShow 自选门控 与 BP「自选=叠加层」是否一致？
- **涉及**：8因子 / 10因子 / 极难=12 / 拯救 / 非酋（全模式横切）
- **背景**：当前 selfShow 门控 = 拯救/随机/极难①/非酋 无自选区（R3 已确认）；标准 8/10/12 有自选区。但 BP 二选一的「自选 1 指挥官」是独立 UI 动作。两套「自选」语义需统一：
  - selfShow 自选区 = 叠加层（拖拽额外指挥官覆盖随机），R3「自选=叠加层」。
  - BP 自选 1 指挥官 = BP 阶段二选一的一条路径。
- **冲突点**：非酋/拯救 selfShow=false，但若 BP 启用自选分支，需要新 UI 路径或解除门控。
- **候选**：
  - A. 两者解耦：selfShow 门控只管叠加层；BP 自选 1 指挥官走 BP 面板独立路径，不受 selfShow 影响（非酋/拯救 BP 自选分支另开 UI）。
  - B. 复用门控：selfShow=false 的模式，BP 自选分支也禁用（拯救随 O4=A 已禁；非酋同理只开 ban）。
- **推荐**：**B**——与 O4 一致、实现最省：selfShow=false 的模式 BP 只开「ban 因子」。但需用户确认非酋是否要破例支持自选指挥官。

### O6. 降权 0.25 落地时对非酋是否生效？
- **涉及**：非酋（兼标准模式落地确认）
- **背景**：降权作用域 = 标准 8/10/12 + 随机；拯救/双打已明确不适用。非酋是单刷 A/B 路径，理论该适用，但非酋已对 A/B 池做特判删人（雷诺/斯台特曼）。
- **候选**：A. 非酋纳入降权（B 组德哈卡/泰凯斯 0.25）；B. 非酋排除。
- **推荐**：**A（统一全单刷模式应用权重）**，countA/countB 可配性允许后期微调。

## P2 · 文档化 / 低风险确认

- **候选**：A. 有意保留；B. 改必然删或不删。

### O8. smallRate=0.9 注释（疑死变量）是否应删？

- **背景**：代码不传 smallRate，等概率分布；注释提及 0.9 折扣未消费。真相表 Q2 标记待确认。

### O9. 非酋删雷诺/斯台特曼来源？3 固定因子来源？不删进攻部署原因？

- **候选**：有意（"非酋"寓意选人倒霉/增难）则文档化到 rules-decisions.md；PoC 遗留则清理。
- **推荐**：确认意图后文档化或删除；3 固定因子明确「赛前配置固定」还是「每周 CSV」。

### O10. 官突 plus 的 mutators + maps 应配置化还是保持 hardcode？
