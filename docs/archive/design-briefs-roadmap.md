### Brief A — BP 违规提示（遗留①·最高优先）

**背景**：集结杯 select（选择面板）已实现 BP「Ban 1 因子 / 自选 1 指挥官」二选一规则，逻辑完整，只差视觉皮。当前两处是临时占位需正式化：① 因子池每个因子左上角的 **ban 角标**（现写死字符「禁」/「⊘」、灰底）；② 被 ban 因子在池中的态（现仅 `opacity:.45`）；③ **软违规提示** —— match（比赛）态下「被禁因子仍落槽」「禁因子与自选指挥官同时启用」属软违规（提示但不强拦），目前复用通用红色校验 toast `.toastv`，与「硬错误（场次未填满）」视觉无分层，主播/选手分不清「这条只是提醒、仍可开赛」还是「这条挡开赛」。注意：练习态无任何 BP 提示，仅 match 态出现。

**要设计的块（逐块确认）**：
1. **因子 ban 角标 `.bp-ban-toggle`**：因子卡（66px）左上角的圆形 toggle，两态——可禁（默认，中性）/ 已禁（`--lose` 语义）。与右上角已存在的「点金」金色角标 `.gold-toggle` 左右分置不撞色。给出 hover/按下反馈。
2. **被禁因子 池中态**：被 ban 的因子卡整体的「不可选」表达（比 `opacity:.45` 更明确，如斜纹/降饱和 + 锁形微标），仍要能看清是哪个因子。
3. **软违规 banner（与硬错误分层）**：在「比赛开始」按钮左侧，区别于硬错误的两级视觉——**硬错误**=阻断（强 `--lose`，现 `.toastv` 形态保留）；**软违规**=警示但放行（建议 `--accent`/琥珀向，附「仍可开赛/超出比赛规则」字样）。支持多条堆叠计数（沿用 `.toastv-n` 的 `+N`）。

**真实文案/数据**（直接用，勿改写）：
- 软违规文案（match 态实际触发）：`禁因子与自选指挥官二选一：已启用禁因子，自选指挥官不可用（超出比赛规则）` / `第2场因子1已被禁用（超出比赛规则）` / `禁因子与自选指挥官二选一（超出比赛规则）`
- 硬错误文案（阻断）：`每场因子槽必满` 类（现 `.toastv` 已承载）
- 角标字面：可禁态可保留极简标记，已禁态明确「禁用」语义；点金角标并存（金色「金」，右上）。
- 因子卡尺寸：池中 66px，槽内 52px。

**约束**：见通用约束。额外——`.toastv` 通用形态已在设计系统/`v4-r2/v4.css §4.2` 存在（金属切角 9px、minimal 圆角 7px、light 用 `color-mix(... var(--lose) 9%, #fff)`），**软违规 banner 复用这套形状语言只换语义色**，不另起一套；ban 角标与 `.gold-toggle`（20px 圆、`z-index:3`、`stopPropagation`）对称。

### Brief B — 截图导出（P2·轻量增量）

**背景**：集结杯要给主播/选手「一键导出当前对局画面」（snapDOM 引擎，工程侧）。需要在直播横条屏（obs）与结算屏（result）加导出入口与导出反馈，并定义导出帧的品牌处理。导出按钮属操作态，**`?bare=1` 纯采集模式不得出现**（现 obs 控制条已是此规则）。

**要设计的块（逐块确认）**：
1. **导出按钮 `.export-btn`**：放在采集线外控制区（obs 屏返回/判定条同排），与现有 `.ctrl-btn`/`.startbtn` 同体系；图标用线性几何（无 emoji），文案「导出当前画面」。
2. **导出微态**：点击后 3 态——导出中（进度/脉冲）/ 已生成（成功，`--win` 向）/ 失败（`--lose` 向），2 秒自动消（沿用现状 toast 4s 节奏可调）。
3. **（可选）导出帧水印条**：导出图底部一条极简品牌条（赛事名 + 选手 ID + 比分），仅出现在导出产物、不入实时画面；用 `BrandLockup` 同款金色处理。

**真实文案/数据**：按钮「导出当前画面」；成功「已生成 · 可保存」；失败「导出失败 · 重试」；水印条示例「集结杯 · 选手 集结杯选手 · 2/3」。

**约束**：见通用约束。按钮形态对齐 `.startbtn`（`--accent` 底、`--on-accent` 字、金属切角）；**采集态隔离**——导出 UI 与 `data-bare` 互斥；水印条用 token 不写死金值。

### Brief C — 码方案 · 生成码面板（P3a·新屏）

**背景**：集结杯「非实时码方案」——主播/裁判先在工具里把一局的选因子结果（地图随机种子 + 各场指挥官 + 因子 + 点金 + ban）固化成一个**可复制的对局码**，发给选手离线对照，避免实时沟通。本屏是「把当前选择导出为码」的面板，repo 无既有设计。模式涵盖 6 种（8/10/极难/拯救/非酋之轮/官突双打），码需承载完整 3 场信息。

**要设计的块（逐块确认）**：
1. **对局摘要卡**：紧凑展示当前局 3 场（地图缩略 + 指挥官头像 + 因子小图 + 点金/锁定标），复用 select/obs 的卡片语言（指挥官 56×67、因子 52）。
2. **码展示区 `.code-box`**：大号等宽 `--font-num` 显示生成的对局码（短码或分段码），右侧「复制」按钮 + 复制成功反馈。
3. **元信息行**：模式名、难度总分、生成时间、选手 ID（沿用 `.topbar-meta` 的 `meta-k/meta-v` 体系）。
4. **底部操作**：复制码 / 重新生成 / 返回，按钮体系对齐 `.startbtn`。

**真实文案/数据**：模式示例「8 因子 · 手选」「官突双打」；难度总分整数（如 `47`）；地图名真实（升格之链 / 机会渺茫 / 往日神庙）；指挥官真实（斯图科夫 / 扎加拉 / 菲尼克斯）；码占位用分段等宽串（如 `JJB-8F-3K9Q-2M7X`）；按钮「复制对局码」/「重新生成」。

**约束**：见通用约束。摘要卡密度对齐 obs 横条（信息饱满不留白）；码区等宽用 `--font-num`；复制成功用 `--win` 文字反馈、无 emoji。

### Brief D — 码方案 · 贴码开局入口（P3b·新块/新屏，与 C 同轮）

**背景**：选手拿到对局码后，在「贴码开局」入口粘贴 → 工具解析 → 预览这局配置 → 一键进入对照。与 Brief C 配对（一出一入）。建议作为首页练习态的一个入口，或独立屏。

**要设计的块（逐块确认）**：
1. **贴码输入 `.code-input`**：单行/分段输入框 + 「解析」按钮，沿用首页 `.player-field`（54px 高、`--panel-bg` 底、`--panel-edge` 边、金色 `--accent` 光标）的输入体系。
2. **解析结果预览**：解析成功后展示该局 3 场摘要（复用 Brief C 摘要卡）；解析失败的错误态（码无效 / 版本不符），用软提示 banner（对齐 Brief A 软违规 banner 语言）。
3. **进入按钮**：「按此码开局」，`.startbtn` 体系。

**真实文案/数据**：占位「粘贴对局码…」；按钮「解析」/「按此码开局」；失败「对局码无效，请核对」「码版本不符」。

**约束**：见通用约束。输入框沿用首页输入语言（不另造）；预览卡与 Brief C 同款保持一致；错误用文字+几何点不用 emoji。

### Brief E — 主播横条 · 选手 ID 强化展示（P4·增量）

**背景**：直播采集横条（ObsBar，1280×232）已含品牌字标、比分（已胜场/总场 + pips）、难度总分、3 场列（地图/指挥官/因子/敌方徽/判定徽）。P4 要在横条上**强化选手 ID 直播展示**——当前横条无独立的「选手名牌」区，选手是谁在画面上不突出。需在不破坏现有信息密度、不超 232px 高的前提下，加一块选手 ID 名牌。

**要设计的块（逐块确认）**：
1. **选手 ID 名牌**：横条左侧比分区附近的选手身份块（选手 ID + LIVE 直播态点 `.pl-dot` 同款），金色 `--accent` 强调，OBS 远距可读（字号偏大）。给出长 ID 截断策略（maxLength 24）。
2. **与比分区的排布关系**：名牌 / 比分 / pips / 难度 四者在左栏的层次（现 `.obs-score` 区），不挤压右侧 3 场列。
3. **直播态指示**：LIVE 点 + （可选）赛季/轮次小标，沿用现状 `mode-light` 适配。

**真实文案/数据**：选手 ID「集结杯选手」（默认值，可长可短）；比分「2 / 3」；难度「47」；LIVE 文案沿用首页 `.player-live` 的 `LIVE`。

**约束**：见通用约束 + **横条专属**——总高 ≤232px、`?bare=1` 采集态必须成立、信息密度对齐现有 obs-fuller（不留白）；名牌用 `--accent`，明暗皮皆远距可读；不挤压 3 场列宽（live 场 hero 加宽逻辑保留）。

### Brief F — 主播登录界面（P5·新屏）

**背景**：比赛态首页现有「主播登录」按钮 `.loginbtn`，状态是占位「即将上线」。P5 后端（PocketBase）落地后需要真实登录界面：主播/裁判账号登录，登录后才可操作判定、生成对局码、写积分。本屏是登录交互全屏（或弹层）。后端接 nginx `/api` 同源反代。

**要设计的块（逐块确认）**：
1. **登录卡**：账号 + 密码（或令牌）输入，沿用首页 `.player-field` 输入语言；主按钮「登录」`.startbtn` 体系。
2. **品牌头 + 角色说明**：`BrandLockup` + 一行「主播 / 裁判后台」副标（沿用 `.match-hint` 体系，金色点 `.mh-dot`）。
3. **状态态**：登录中 / 失败（账号或密码错）/ 成功跳转，错误用软 banner（对齐 Brief A）。
4. **次要入口**：返回练习 / 忘记密码（弱化，`--muted`）。

**真实文案/数据**：标题「主播登录」；副标「ADMIN · 比赛后台」；占位「主播账号」「密码」；按钮「登录」；失败「账号或密码错误」；现状占位文案 `ADMIN · 即将上线` 可作为未接通态保留。

**约束**：见通用约束。登录卡居中于 1280×720；输入/按钮全部复用现有体系不另造；无 emoji，状态用文字。

### Brief G — 积分天梯页（P6·新屏）

**背景**：比赛态首页现有「积分天梯」占位卡 `.pcard.ladder.soon`（即将上线，副标「赛季积分 · 全国排名 · 段位结算」）。P6 后端 hook 落地后需真实天梯页：赛季积分榜、选手排名、段位。本屏是天梯全屏展示（直播也可投）。

**要设计的块（逐块确认）**：
1. **天梯榜列表**：每行=名次 + 选手 ID + 积分 + 段位 + 战绩（胜/总），复用 BpConfigScreen 的 `.bpc-row` 行卡语言（左 edge 条 + no + main + 右值）。前三名强调（金/银/铜向，金用 `--accent`）。
2. **赛季头**：赛季名 + 周期 + 我的排名高亮（沿用 `.topbar-meta`）。
3. **段位/积分图例**：底部 legend（沿用 `.bpc-legend` 体系）。
4. **空态/加载态**：未接后端时的占位（对齐现状 `soon-chip`「即将上线」）。

**真实文案/数据**：副标沿用「赛季积分 · 全国排名 · 段位结算」；行示例「01 · 集结杯选手 · 1280 分 · 王者 · 18/20」；名次用 `--font-num`；未接态「即将上线」。

**约束**：见通用约束。行卡复用 `.bpc-row` 不另造表格；前三名强调用 token；可投直播 → 远距可读、明暗皮成立；无 emoji，名次/段位用文字+几何。
